## Mapping notes: vendor payloads → `AdminAnalyticsPayload`/`AdminAnalyticsRow`

Target shape (`src/lib/admin-analytics.ts`): `{ period, season, market, employeeId, salesVolume, rentalRevenue, occupiedNights, availableNights, closedDeals }`, monthly grain, per market + per employee.

| Field | PriceLabs `get_listing_performance_metrics` | Beyond Partners API | Wheelhouse RM API | Hostaway finance reports | Metabase |
| --- | --- | --- | --- | --- | --- |
| `period` | DFD (days-from-date) keys, not calendar months — needs a remap from relative-day buckets to `YYYY-MM-01` | Calendar-based via listing/calendar endpoints | "Rolling-window" + monthly/quarterly/yearly stats — closest native fit | `fromDate`/`toDate` range per request, arbitrary — you choose the month boundary | N/A (BI tool queries whatever warehouse schema you give it) |
| `season` | Not provided — derive client-side from month, same as `admin-analytics.ts`'s `seasonForMonth()` already does | Not provided — derive | Not provided — derive | Not provided — derive | Derive in the SQL model if warehouse has dates |
| `market` | No native "market" grouping in this endpoint — it's per-`listing_id`; market-level comps come from a *separate* `get_neighbourhood_data`/`get_str_index` call, not the same shape | No native grouping — per listing/property | Portfolio-level stats exist, per-listing detail — "market" is an OSEAD-side aggregation over listing→market lookup you'd maintain yourself | `listingMapIds[]` param — same problem, no vendor-side market rollup | Whatever your warehouse's market column is — this is the one integration where "market" is free if your schema already has it |
| `employeeId` | **No concept of employee/agent at all** — PriceLabs/Wheelhouse/Beyond are all revenue-management tools for *listings*, not agent-attribution systems | Same — no agent/employee field | Same — no agent/employee field | Same — Hostaway has no "sales agent" concept (it's a PMS, not a CRM) | Depends entirely on whether your warehouse joins reservations to a staff/agent table |
| `salesVolume` (closed **property sales** $) | **Not applicable** — PriceLabs/Beyond/Wheelhouse/Hostaway are all STR/rental tools; none touch property *sales* transactions at all. This field can only ever be filled by a brokerage CRM (kvCORE, Follow Up Boss, BoomTown, Sierra) via their transactions endpoints, none of which were in this pass's top-5 | — | — | — | If a broker's warehouse has closed-sale records, trivially mappable |
| `rentalRevenue` | `listing_level.revenue[dfd]` — direct fit, needs DFD→month remap | Listing/calendar pricing data — revenue would need to be derived from booked calendar entries × rate, not a single revenue field found in the docs reviewed | RM API "KPIs" explicitly include revenue rollups — direct fit | `finance/report/listingFinancials` gives line-item accommodation revenue — sum per listing/month yourself | Direct, if warehouse has it |
| `occupiedNights` / `availableNights` | `listing_level.occupancy[dfd]` is a **rate**, not raw night counts — you'd need `occupancy × available_nights_in_period` to back into a night count, and available nights isn't directly returned either (derive from calendar length) | Calendar endpoint gives day-by-day availability/booked state — nights are countable by iterating the calendar array | RM API KPIs include occupancy — same rate-vs-count caveat as PriceLabs | Occupancy Report formula is `nightsBooked * 100 / totalNightsAvailable` — Hostaway's own UI computes this from calendar+reservation data, same raw ingredients OSEAD would need via `/v1/listings/{id}/calendar` + `/v1/reservations` | Direct, if warehouse has it |
| `closedDeals` | **Not applicable** (see `salesVolume`) | — | — | — | Brokerage-CRM concept only |

**Bottom line for the mapping:** none of the 5 spot-checked platforms can fill `AdminAnalyticsRow` end-to-end from a single endpoint. All of them are missing `market` (as an OSEAD-defined grouping) and `employeeId`/`salesVolume`/`closedDeals` (brokerage-CRM-only concepts). PriceLabs and Wheelhouse get closest on the *rental* half of the row (`rentalRevenue`, `occupiedNights`/`availableNights` via rate math); a real integration would need a small server-side adapter that (a) calls the vendor API per listing, (b) joins against an OSEAD-side listing→market and listing→agent lookup table, and (c) leaves `salesVolume`/`closedDeals` at 0 unless a CRM is also wired in. This confirms the source doc's own conclusion (§"What OSEAD should own first-party") — the rollup shape is OSEAD's IP, not something any vendor hands you pre-built.

## Import/export API provisions (white-label / broker-facing)

The brief asks for two directions. `admin-analytics.ts` already defines the shape; this section documents the provisions, not new code.

### Outbound: OSEAD → broker's own dashboard/BI tool

A broker who wants to pull OSEAD-shaped data into their *own* Metabase/warehouse/spreadsheet needs a stable JSON contract to poll. That contract already exists as the `AdminAnalyticsPayload` TypeScript interface — the only missing piece is *publishing* it, not defining it:

- **Shape:** `{ source: "demo" | "external", generatedAt: ISO8601, currency: "USD", rows: AdminAnalyticsRow[] }`, documented in full at `src/lib/admin-analytics.ts:13-33`.
- **Today:** this shape is only ever consumed client-side inside the Svelte island (`loadAdminAnalytics()`); there is no route that *serves* it back out.
- **What "implement now" looks like:** a static Cloudflare Pages Function (e.g. `functions/api/admin-analytics.json.ts`) that calls `buildDemoAdminAnalytics()` (or reads a committed JSON fixture) and returns it with `Content-Type: application/json` — this is the exact shape a broker's Metabase/warehouse ETL job or a competitor's dashboard could poll. Zero new data model, zero vendor keys, reuses the existing seed function verbatim.
- **Auth:** none needed for a demo; a real deployment would gate this behind the same secret/token pattern used for any other read-only reporting API (bearer token checked in the Pages Function, not in `admin-analytics.ts` itself).

### Inbound: broker's PMS/CRM → OSEAD

Already fully specified and shipped in `admin-analytics.ts` — this pass confirms the two documented paths are complete and don't need new code, only documentation:

1. **`window.__OSEAD_ADMIN_ANALYTICS__`** — synchronous global set before the Svelte island mounts. Best for a parent CMS/shell that already fetched+reshaped vendor data server-side (e.g. a Next.js/Astro SSR wrapper embedding the OSEAD admin island). `parseAdminAnalytics()` validates shape and coerces `source` to `"external"`.
2. **`PUBLIC_ADMIN_ANALYTICS_URL`** — REST poll, `fetch(url, { headers: { Accept: "application/json" } })`, same validation/coercion. This is the target for the Cloudflare Pages Function pattern described in the source doc's Integration Patterns table (vendor webhook → Pages Function → re-served JSON at this URL).

**What's missing (documentation gap, not code gap):** there is no example adapter script/fixture showing a real vendor payload (e.g. a Hostaway `finance/report/listingFinancials` CSV or a PriceLabs `listing_metrics` JSON) transformed into one `AdminAnalyticsRow`. This is the single highest-leverage doc-only artifact to add next — see "build now" list below.
