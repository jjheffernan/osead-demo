# Validation: presentation/chart research vs. the real codebase

Secondary-swarm stress test of [`admin-dashboard-presentation.md`](./admin-dashboard-presentation.md) against `AdminAnalytics.svelte`, `AdminShell.svelte`, `package.json`, `src/registry.json`, and `src/components/ui/*`. Scope matches the source doc: presentation only, no data-contract changes.

## Verdict

**Stay zero-deps.** The ladder's conclusion is right and the codebase backs it up — but the "proposed architecture" section has two implementation claims that don't survive contact with the framework boundary. Both are fixable without adding a dependency; neither changes the overall recommendation.

1. `Table.astro` and `Skeleton.astro` **cannot be imported into `AdminAnalytics.svelte`.** Astro components only render inside `.astro` templates; they are not consumable from Svelte/React islands. Grepped the whole `src/components` tree for `.astro` imports inside `.svelte` files — zero hits, no precedent, no supported path. The proposed swap ("Employee leaderboard → Table.astro", "Skeleton.astro for the loading state") has to become either a hand-rolled `<table>`/skeleton in `AdminAnalytics.svelte`'s own `<style>` block (what it already does today) or the Table/Skeleton *markup pattern* copied by hand — not a cross-framework import.
2. **`client:visible` cannot be attached to `TrendChart.svelte` as drawn.** `client:*` directives are an Astro→framework hydration boundary; they only apply where Astro mounts a top-level component. `TrendChart` in the proposed tree is nested inside `AdminAnalytics.svelte`, itself nested inside `AdminShell.svelte`, which `account.astro` already mounts with a single `client:load`. Everything under that root hydrates together in one JS execution — there is no per-child hydration directive available deeper in the tree. If uPlot is ever added, "mount only when visible" has to be hand-rolled (`IntersectionObserver` in `onMount`, or a dynamic `import("uplot")`), not expressed as a template attribute.

Neither issue is a reason to add a dependency — they're corrections to *how* the zero-dep plan gets built, not to *whether* it should stay zero-dep.

## Hard "do not add" list

- **Chart.js, ECharts, Plotly, Billboard.js, Observable Plot, any D3-peer-dep library** — confirmed zero charting packages currently exist anywhere in `pnpm-lock.yaml`; the data shape (12×3×N rows, filter-driven, not streaming) never grows into needing any of these.
- **React admin dashboard kits (e.g. Tremor, shadcn/admin templates, AdminJS, Refine)** — this is a Svelte 5 + Astro island stack top to bottom; a React kit means shipping a second UI framework's runtime *and* runtime tooling (JSX transform, its own component conventions) inside `AdminShell`. It would fork the design system: OSEAD's tokens are consumed as CSS vars (`var(--border)`, `var(--foreground)`) directly in `<style>` blocks (see every `ui/` primitive and `AdminAnalytics.svelte`); most React admin kits ship Tailwind-config-driven or CSS-module theming that doesn't map 1:1 onto a semantic-token, non-Tailwind-palette convention enforced by `check-kpis.mjs`'s `HARDCODED_TW_COLOR` check. Every card/table/chart from the kit would need re-theming by hand anyway — at which point you've bought a dependency and thrown away its main value (pre-themed components).
- **A second design-token or component convention** — the codebase already has a live inconsistency worth not making worse: `AdminAnalytics.svelte` uses canonical token names (`--border`, `--muted-foreground`, `--spacing-5`, `--text-xs`) directly, while `ui/` primitives (`Card.svelte`, `Table.astro`, `Tabs.svelte`) use a legacy alias layer (`--color-border`, `--space-lg`) that resolves back to the same tokens via `colors.css`/`spacing.css` (verified — the aliases are real and functional, not dead). Both work today, but a React kit or a Chart.js wrapper would add a *third* theming convention (component-library props/theme objects) on top of two that already coexist. Any new admin widget should pick the canonical token names (matches the file it's extending) rather than reintroduce the legacy alias layer or invent a new one.
- **TanStack Table / any headless table library** — already correctly rejected by the source doc; confirmed the actual table is 8 rows, no sorting/pagination/virtualization requirement exists in `admin-analytics.ts`.
- **BI iframe embeds (Looker Studio/Metabase/Grafana) as a default** — confirmed `public/_headers`' CSP would need a `frame-src` loosening not currently present; and confirmed the entire admin surface is a client-only, localStorage-gated demo (`account.astro`'s `data-account-admin hidden`, revealed by an inline script checking `readDemoAccount()`) with no real backend session — wiring a real BI vendor in as the default would contradict "Demo seed by default" messaging already printed in the component's own `<p class="admin-analytics__lede">`.
- **`jsdom`/`linkedom` for server-rendering Observable Plot** — correctly identified by the source doc as a real option, correctly not recommended; would be a build/runtime dependency addition for a benefit (SSR of exploratory-chart markup) this repo's data shape doesn't need.

## Unused-JS / SSR-safety notes (with one correction)

- **The entire admin dashboard is already client-only, not progressively enhanced.** `account.astro` renders `<div data-account-admin hidden>` — there is no server-rendered admin content; it's revealed by an inline `<script>` after checking `localStorage` for a demo account, and `AdminShell` itself mounts with `client:load` (eager, whole-tree hydration, confirmed in `account.astro`). This matters for how much weight "SSR-friendly" claims should carry here: native SVG/CSS is still the right choice (zero deps, direct token theming, a11y control, smaller diff), but *not* because it meaningfully improves this route's no-JS story — there effectively isn't one. Don't oversell "SSR-safe" as a progressive-enhancement win for this specific gated surface; it's a code-simplicity and bundle-size win instead.
- **`client:visible` cannot gate a nested Svelte child** (detailed in Verdict above). Concretely: `AdminShell` (`client:load`) → `AdminAnalytics` → hypothetical `TrendChart` are all one hydration unit today. If uPlot is added later, "only pay canvas cost when the Analytics panel is actually open" already happens for free via `AdminShell`'s `{#if panel === "analytics"}` (Svelte doesn't mount components inside a false `{#if}` branch, so `TrendChart` wouldn't even construct until a user clicks into Analytics) — but "only pay canvas cost when scrolled into view *within* the open panel" needs a manual `IntersectionObserver` in `TrendChart`'s own `onMount`, not a template directive.
- **Everything proposed in the widget list above is SSR-safe by construction** (plain markup, no canvas) except the future uPlot rung, which the source doc already correctly scopes to "ship a static placeholder/skeleton for SSR" — consistent with the existing `<table>`-as-fallback pattern once a loading-state branch exists to hang it on.

## Suggested order of UI slices for AFK build

Ordered so each slice is independently buildable/testable (matches the vertical-slice model in `admin-analytics-feature.md`), cheapest/lowest-risk first:

1. **Sales trend sparkline** in `AdminAnalytics.svelte` — new `SalesSparkline.svelte`, pure `$derived` + `<svg><polyline>`, no shell/data changes. Highest value-to-risk: it's the one visual gap rung 2 doesn't already cover in this file.
2. **Employee leaderboard bars** — CSS-only change to the existing `<table>` in `AdminAnalytics.svelte` (per-row `%`-width bar, same technique as occupancy). No new component.
3. **Market split stat row** — native 3-tile row, same markup pattern as the existing KPI strip. No new component.
4. **Badge swap** for the source/demo-seed pill — mechanical, low-risk, already-importable `.svelte` component.
5. **Loading-state branch** — add the missing `{#if status === "loading"}` case with hand-rolled shimmer `<div>`s (copy `Skeleton.astro`'s CSS, don't import it).
6. **Overview KPI strip** — independent `loadAdminAnalytics()` call from the `panel === "overview"` branch in `AdminShell.svelte`, rendering a condensed 3-KPI subset next to the existing `.admin-panel__stats` inventory tiles. Slightly more risk (touches `AdminShell`, a second data fetch) — sequence after the Analytics-panel slices are proven.
7. **KPI tile → `Card.svelte`** — hold this for a deliberate design pass, not an AFK auto-slice: it changes visual language (boxed cards vs. hairline grid) and should be a single reviewed decision, not silently absorbed into another ticket.
8. **uPlot trend chart** — do not schedule into this AFK run at all. No current ticket/data shape justifies rung 3; revisit only if a future ticket explicitly asks for multi-series time-series with zoom.
